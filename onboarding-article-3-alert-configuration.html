Alert Threshold Settings

Prevent false alarms and alert fatigue

üö® The Problem with Instant Alerts

If you alert on the first failed check, you'll get false alarms from temporary network blips, brief server restarts, and random internet hiccups. This leads to alert fatigue (you start ignoring them).

Understanding Thresholds

What Are Thresholds?

A threshold is how many consecutive failed checks must occur before an alert is sent. Think of it as a filter for false positives.

Example: If threshold = 2, and check interval = 5 minutes:
1st check fails (0 minutes) ‚Üí No alert
2nd check fails (5 minutes) ‚Üí Alert sent
Total time before alert: 5 minutes

Recommended Threshold Settings

‚úì Best for Most Cases: 2 Checks

Filters out random blips but still alerts quickly. With 5-minute intervals, you're notified within 5-10 minutes of real downtime.

3 Checks (Conservative)

Use for less critical monitors or if you're getting too many false positives. Takes 10-15 minutes to alert.

1 Check (Instant, Not Recommended)

Only use for EXTREMELY critical systems where every second counts. Expect false alarms.

How to Configure

You can set this per monitor or use the default for all:

  • ‚Ä¢ Edit monitor ‚Üí Advanced Settings
  • ‚Ä¢ Look for "Alert threshold"
  • ‚Ä¢ Set to 2 (recommended default)
  • ‚Ä¢ Save

When to Get Alerted

Choose your notification triggers

! Down Alerts (ALWAYS Enable)

Get notified when your site goes down. This is non-negotiable.

Default: Enabled for all monitors
Action: Keep this on. Always.

‚úì Up Alerts (Recommended)

Get notified when your site comes back online. Confirms the issue is resolved.

Why useful: Know when to stop panicking. Especially helpful at 3 AM.

Action: Enable for all SSL monitors. Set threshold to 7+ days.

Avoiding Alert Fatigue

Stay responsive without getting overwhelmed

‚ö†Ô∏è What Is Alert Fatigue?

When you get so many alerts that you start ignoring them. This defeats the entire purpose of monitoring. You'll miss real emergencies because you're drowning in noise.

Prevention Strategies

1. Don't Over-Monitor

More monitors ‚â† better. Only monitor what matters.

‚úì Do Monitor:

  • ‚Ä¢ Revenue-generating pages
  • ‚Ä¢ Critical user flows
  • ‚Ä¢ Authentication systems

‚úó Don't Monitor:

  • ‚Ä¢ Every single blog post
  • ‚Ä¢ Static pages that rarely change
  • ‚Ä¢ Internal admin pages (unless revenue-critical)

2. Group Alerts Logically

Use different alert contacts for different priority levels:

  • Critical (SMS + Email): Checkout, login, main systems
  • Important (Email only): Product pages, customer-facing features
  • Low Priority (Email to team channel): Nice-to-have pages

3. Set Up Maintenance Windows (Pro Feature)

Schedule planned maintenance? Silence alerts during these windows so you don't get false alarms.

Example: Server updates every Sunday 2-4 AM? Create a maintenance window. No alerts during this time.

4. Weekly Alert Review

Every week, review your alerts:

  • ‚Ä¢ Are you getting too many false positives? Increase threshold.
  • ‚Ä¢ Getting alerts for pages that don't matter? Remove monitors.
  • ‚Ä¢ Not getting critical alerts? Check contact verification.
  • ‚Ä¢ Adjust and optimize continuously

üö® CRITICAL: Always Test!

"I set up alerts" means nothing until you've verified they actually reach you. Test now, while there's no emergency.

How to Test Alerts

Method 1: Use Test Feature (Easiest)

  • 2. Click "Test Alert"
  • 3. Check email/SMS/Slack

Method 2: Create a Test Monitor

Monitor a URL you control and can break temporarily:

  • 1. Create a simple HTML page on your site: /test-alert.html
  • 2. Add it as a monitor in UptimeRobot
  • 3. Delete or rename the page
  • 4. Wait for alert (should come within 5-10 minutes)
  • 5. Restore the page
  • 6. Verify you get "Up" alert

If you're not receiving alerts:

  • ‚Ä¢ Check spam/junk folder
  • ‚Ä¢ Add UptimeRobot to safe senders
  • ‚Ä¢ Whitelist: alerts@uptimerobot.com
